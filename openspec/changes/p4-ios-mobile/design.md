# P4 iOS 移动端适配 - 技术设计

## 1. 技术选型

### 1.1 总体方案：Capacitor + React + Vite

选择 Capacitor 而非 React Native 或 SwiftUI 的原因：

| 方案 | 代码复用率 | 开发成本 | 性能 | 维护成本 |
|------|-----------|---------|------|---------|
| **Capacitor** | 80%+ | 低 | 中 | 低（一套代码） |
| React Native | 20% | 高 | 高 | 高（两套代码） |
| SwiftUI | 0% | 极高 | 最高 | 极高（三套代码） |

**结论**：Capacitor 最符合当前阶段需求——保留现有 React 代码和业务逻辑，只需重构 UI 层。

### 1.2 技术栈对比

| 层级 | 桌面端（当前） | 移动端（目标） |
|------|---------------|---------------|
| 框架 | React 19 + Vite 8 | React 19 + Vite 8 |
| UI 组件 | Ant Design 6 | Ant Design Mobile 5 |
| 画布 | Konva (react-konva) | Konva (react-konva) + 手势适配 |
| 路由 | react-router-dom v7 | react-router-dom v7 |
| 打包 | Electron + electron-builder | Capacitor + Xcode |
| 后端 | 本地 Python FastAPI | 云端 FastAPI + 本地 SQLite 回退 |
| 数据库 | 本地 SQLite | 云端 SQLite / PostgreSQL + 本地 SQLite |
| 存储 | 本地文件系统 | 本地沙盒 + iCloud |

### 3.3 拼贴编辑器移动端改造

- 画布使用鼠标拖拽绘制，不支持触控

- 属性面板固定右侧，手机无法容纳
- 右键菜单无法在手机上触发

**改造方案**：

```
┌─────────────────────────┐

│                         │
```

## 4. 后端适配

### 4.1 当前问题

Python FastAPI 本地服务无法在 iOS 上运行。需要将后端拆分为：

1. **云端 API 服务**：处理抠图、素材存储等需要服务器能力的功能
2. **本地存储**：使用 Capacitor 的 Filesystem + SQLite 插件实现离线模式

### 4.2 云端部署方案

```
┌──────────────────────────────────────────┐
│              云端服务器                    │
│  ┌────────────────────────────────────┐  │
│  │  FastAPI (现有代码，几乎不变)         │  │
│  │  - /api/categories                  │  │
│  │  - /api/materials                   │  │
│  │  - /api/backgrounds                 │  │
│  │  - /api/collages                    │  │
│  │  - /api/materials/upload            │  │
│  │  - /api/materials/{id}/removal      │  │
│  └────────────────────────────────────┘  │
│  ┌────────────────────────────────────┐  │
│  │  PostgreSQL / SQLite                │  │
│  │  + 文件存储（本地磁盘 / OSS）        │  │
│  └────────────────────────────────────┘  │
└──────────────────────────────────────────┘
         │
         │ HTTPS
         │
┌────────┴───────────────┐
│     iOS App            │
│  ┌──────────────────┐  │
│  │  api.ts           │  │
│  │  baseURL = cloud  │  │
│  └──────────────────┘  │
│  ┌──────────────────┐  │
│  │  本地 SQLite      │  │  ← 离线缓存
│  │  (capacitor-      │  │
│  │   community/sqlite)│  │
│  └──────────────────┘  │
└────────────────────────┘
```

### 4.3 离线模式策略

```
用户操作 → 先写本地 SQLite → 网络可用时同步到云端
         → 读取时优先本地缓存，后台刷新云端数据
```

## 5. Capacitor 配置

### 5.1 项目初始化

```bash
npm install @capacitor/core @capacitor/cli @capacitor/ios
npx cap init "手账素材库" "com.sticker.material" --web-dir=dist
npx cap add ios
```

### 5.2 需要的插件

| 插件 | 用途 |
|------|------|
| `@capacitor/camera` | 拍照上传素材 |
| `@capacitor/filesystem` | 本地文件存储 |
| `@capacitor/share` | 分享拼贴作品 |
| `@capacitor/haptics` | 触觉反馈 |
| `@capacitor/status-bar` | 状态栏控制 |
| `@capacitor-community/sqlite` | 本地数据库 |
| `@capacitor/ios` | iOS 平台支持 |

### 5.3 构建流程

```bash
# 开发
npm run dev              # Vite 开发服务器
npx cap sync ios         # 同步到 iOS 项目
npx cap open ios         # 在 Xcode 中打开

# 生产构建
npm run build            # Vite 构建
npx cap sync ios         # 同步
# Xcode → Product → Archive → Distribute
```

## 6. 关键技术难点

### 6.1 Konva 触控适配

- Konva 本身支持 touch 事件，但需要正确配置 `Stage` 的 touch 属性
- 需要禁用 WebView 的默认手势（如双击缩放、边缘滑动返回）
- 捏合缩放的坐标转换需要特别注意

### 6.2 图片处理

- 桌面端 `rembg` 抠图依赖 Python，移动端无法本地运行
- 方案：抠图请求发送到云端处理，本地仅展示结果
- 用户拍照后先上传到云端，云端处理完返回抠图结果

### 6.3 性能优化

- 大量素材时使用虚拟列表
- 拼贴画布限制最大元素数量（建议 50 个）
- 图片懒加载 + 缩略图
- WebView 内存管理（iOS WKWebView 限制）

### 6.4 屏幕适配

- 支持 iPhone SE (375pt) 到 iPhone Pro Max (430pt)
- 画布预设尺寸适配手机屏幕比例
- Safe Area 适配（刘海屏、灵动岛）

## 8. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| Konva 触控性能不达标 | 拼贴编辑器卡顿 | 降低画布分辨率、限制元素数量、使用 WebGL 渲染 |
| 云端服务不稳定 | 素材无法加载 | 本地 SQLite 离线缓存，网络恢复后自动同步 |
| App Store 审核被拒 | 无法上架 | 确保隐私合规、不使用私有 API、提供完整隐私政策 |
| 图片上传流量大 | 用户流量消耗 | Wi-Fi 下自动上传，移动网络提示确认 |
| 抠图服务成本高 | 运营成本 | 限流、排队、VIP 用户优先 |
